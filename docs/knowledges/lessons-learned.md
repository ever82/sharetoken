## 二、走过的弯路

### 弯路 #1: Issue #1 CI/CD的grep模式匹配问题

**问题**: 测试脚本test_cicd.sh使用grep匹配命令输出，但CI实际使用的是`make test`而非直接`go test`

**浪费的时间**: ~30分钟调试grep模式

**更优方案**:
- 不应依赖命令行输出的文本匹配
- 应直接检查CI配置文件是否存在且语法正确
- 或运行轻量级验证而非完整模拟CI

**经验教训**: 验收测试不应过度依赖文本匹配，应检查实质配置。

### 弯路 #2: Issue #2 UPnP实现初期的方案选择

**问题**: 一开始不确定UPnP应该简单实现还是完整实现，导致反复询问用户

**浪费的时间**: ~15分钟沟通成本

**更优方案**:
- 直接提供两种方案的技术对比和推荐
- 让用户快速决策

**经验教训**: 技术方案应附带明确的推荐选项，减少决策负担。

### 弯路 #3: Issue #2 P2P消息广播测试

**问题**: for-dev.md要求"1000条消息广播无丢失"，实际未进行此测试

**浪费的时间**: ~10分钟讨论是否需要实现

**更优方案**:
- 初期直接标记为"需要压力测试环境，延后验证"
- 不浪费时间在无法本地验证的测试上

**经验教训**: 对于需要特殊环境的测试，应及时延后而非尝试在本地实现。

### 弯路 #4: 多节点网络启动脚本的反复修复

**问题**: devnet_multi.sh经历了多次修复：
1. 端口冲突
2. 节点数据目录问题
3. 配置格式问题

**浪费的时间**: ~1小时

**更优方案**:
- 初期应使用Ignite CLI的network命令而非手动编写脚本
- 或使用docker-compose统一管理

**经验教训**: 复用现有工具比从零编写脚本更可靠。

### 弯路 #5: Makefile中的ignite命令冲突

**问题**: React Native的ignite与Cosmos SDK的Ignite CLI同名冲突

**浪费的时间**: ~20分钟排查

**更优方案**:
- 早期就应检查PATH中是否有同名命令
- 在Makefile中显式指定完整路径

**经验教训**: 环境检查应在开发早期进行。

## 三、踩过的坑

### 坑 #1: GitHub Actions工作流无法本地完全验证

**症状**: CI配置看起来正确，但GitHub上运行可能失败

**影响**: 需要push到GitHub才能验证，迭代慢

**解决方案**:
- 使用act工具本地运行GitHub Actions
- 或创建最小化测试先行验证

**应记录到文档**: 开发流程应包含act工具的使用

### 坑 #2: UPnP测试需要真实路由器环境

**症状**: 代码编译通过，但功能无法在本地验证

**影响**: 无法确认代码是否真的工作

**解决方案**:
- 代码完成后先标记"实现完成"
- 真实环境测试通过后更新为"测试通过"
- 建立postponed.md记录延后项

**应记录到文档**: 哪些测试必须延后，如何判断

### 坑 #3: 前端代码与CLI功能不同步

**症状**: CLI转账测试通过，但前端可能使用不同API

**影响**: 可能出现CLI工作但前端不工作的情况

**解决方案**:
- 统一使用CosmJS库
- 前端和CLI使用同一套API端点

**应记录到文档**: 前端与CLI应共享相同的API配置

### 坑 #4: devnet目录数据残留导致问题

**症状**: 多次启动开发网络后，旧数据导致新启动失败

**影响**: 需要手动清理.devnet目录

**解决方案**:
- 启动脚本添加清理逻辑
- 或使用临时目录

**应记录到文档**: 开发网络使用须知

### 坑 #5: Git提交包含不必要的文件

**症状**: .devnet目录的日志和数据文件被提交

**影响**: 仓库体积膨胀，git历史混乱

**解决方案**:
- 立即更新.gitignore
- 使用git filter-branch清理历史（如需要）

**应记录到文档**: 哪些文件不应提交

## 四、正确的决策

### 决策 #1: 延后区块浏览器和前端运行时测试 ✅

**理由**: 这些功能不影响核心网络功能，且需要部署环境

**结果**: Issue #2/#3能够顺利完结，不被阻塞

### 决策 #2: 使用goupnp库而非自行实现UPnP ✅

**理由**: UPnP协议复杂，成熟库更可靠

**结果**: 500+行代码实现了完整的UPnP功能

### 决策 #3: CLI先行验证，前端延后 ✅

**理由**: CLI测试更可靠，不依赖浏览器环境

**结果**: Issue #3核心功能快速验证完成

### 决策 #4: 创建done目录区分已完成任务 ✅

**理由**: 清晰区分进行中和已完成任务

**结果**: 项目状态一目了然

### 决策 #5: 使用CC-NOTIFY通知机制 ✅

**理由**: 用户能及时了解关键进展和阻塞

**结果**: 沟通顺畅，用户满意

## 七、给后续AI开发的建议

### 开始新Issue前

1. **检查环境**: 运行环境检查清单
2. **阅读lessons-learned.md**: 了解之前的坑
3. **确认验收标准**: 与for-dev.md对比，识别可能延后的项
4. **检查postponed.md**: 了解当前延后项状态

### 开发过程中

1. **及时记录问题**: 发现问题立即记录到issue文档
2. **区分代码完成和测试完成**: 两者分开验收
3. **遇到阻塞立即通知**: 使用CC-NOTIFY
4. **定期同步CLAUDE.md**: 保持项目状态最新

### 完成Issue时

1. **更新所有相关文档**: for-dev.md, CLAUDE.md, issue文档
2. **移动到done目录**: 物理区分已完成任务
3. **关闭GitHub Issue**: 添加完整评论
4. **检查postponed.md**: 确认是否有相关项可以更新
